لماذا Next.js • File-based Routing • Server مقابل Client Components • SSR وSSG

🎓 أهلاً بعالم Next.js!

حتى الآن بنينا كل تطبيقاتنا بـ React وVite — قوي ومرن، لكنه يتطلّب إعداد كل شيء بأنفسنا: Routing، تحسين الأداء، حتى SEO.

Next.js إطار عمل (Framework) مبني فوق React نفسه، يمنحك كل هذه الأشياء جاهزة افتراضياً. اليوم نأخذ أول خطواتنا فيه.

1.1

React مكتبة، Next.js إطار عمل

React نفسه مكتبة (Library) تُركِّز فقط على بناء الواجهات — كل شيء آخر (التنقّل، الخادم، الأداء) تختاره وتُعِدُّه بنفسك. Next.js إطار عمل (Framework) متكامل يبني فوق React ويوفّر كل هذه الأشياء جاهزة بقرارات مدروسة مسبقاً.

الجزء الثالث: File-based Routing

في Next.js، بنية المجلدات داخل app/ تُحدِّد مسارات موقعك مباشرة — بلا أي إعداد Router يدوي كما فعلنا سابقاً بـ react-router-dom.

3.1

البنية والمسارات الناتجة

Shell 
app/
├── page.js                → المسار: /
├── about/
│   └── page.js              → المسار: /about
├── blog/
│   ├── page.js                → المسار: /blog
│   └── [slug]/
│       └── page.js             → المسار: /blog/أي-قيمة (مسار ديناميكي)

الجزء الرابع: Server Components مقابل Client Components

4.1

الافتراضي: Server Component

JSX 
// app/page.js — Server Component افتراضياً، بلا أي تصريح خاص
async function getData() {
  const res = await fetch('https://api.example.com/posts');
  return res.json();
}

export default async function HomePage() {
  const posts = await getData();
  return (
    <div>
      {posts.map((post) => <p key={post.id}>{post.title}</p>)}
    </div>
  );
}
// يُنفَّذ هذا الكود بالكامل على الخادم — لا يصل لمتصفح المستخدم كـ JavaScript إطلاقاً!
4.2

"use client": متى نحتاجه

JSX 
// app/components/Counter.js
'use client'; // ← يُخبر Next.js: هذا المكوّن يحتاج تفاعلاً داخل متصفح المستخدم

import { useState } from 'react';

export default function Counter() {
  const [count, setCount] = useState(0);
  return <button onClick={() => setCount(count + 1)}>{count}</button>;
}
💡 القاعدة

"use client" في أول سطر من الملف يُحوِّل المكوّن (وكل ما يستورده) إلى Client Component. أي مكوّن يستخدم useState أو useEffect أو أحداثاً كـ onClick يحتاج هذا التصريح — لأنها لا تعمل إلا داخل متصفح المستخدم.

الجزء الخامس: أساسيات SSR وSSG

المفهومالفكرة المبسّطةيناسب
SSR (Server-Side Rendering)الصفحة تُبنى على الخادم مع كل طلب، ثم تُرسَل جاهزة للمتصفحمحتوى يتغيّر باستمرار (لوحة تحكّم، بيانات حيّة)
SSG (Static Site Generation)الصفحة تُبنى مرة واحدة وقت النشر، وتُقدَّم كملف جاهز للجميعمحتوى لا يتغيّر كثيراً (مدونة، صفحة "من نحن")
💡 لماذا هذا مهم؟

محركات البحث مثل Google تقرأ محتوى HTML الجاهز مباشرة (تحسين SEO حقيقي)، والمستخدم يرى المحتوى فوراً بلا انتظار تحميل JavaScript أولاً — بعكس React وحده حيث تبدأ الصفحة فارغة تماماً حتى يكتمل تحميل JavaScript وتنفيذه.

الجزء السادس: المشروع — أول تطبيق Next.js

مدونة بسيطة بأربع صفحات: تخطيط عام مشترك، رئيسية، من نحن، وقسم مدونة بمسار ديناميكي لكل مقال.

app/layout.js

JSX 
import './globals.css';

export const metadata = {
  title: 'مدونتي الأولى بـ Next.js',
  description: 'أول تطبيق Next.js كامل',
};

export default function RootLayout({ children }) {
  return (
    <html lang="ar" dir="rtl">
      <body>
        <nav>
          <a href="/">الرئيسية</a>
          <a href="/about">من نحن</a>
          <a href="/blog">المدونة</a>
        </nav>
        <main>{children}</main>
      </body>
    </html>
  );
}

app/blog/page.js وapp/blog/[slug]/page.js

JSX 
// app/blog/page.js
const posts = [
  { slug: 'react-hooks', title: 'مقدمة إلى React Hooks' },
  { slug: 'nextjs-intro', title: 'لماذا Next.js؟' },
];

export default function BlogPage() {
  return (
    <div>
      <h1>📝 المدونة</h1>
      <ul>
        {posts.map((post) => (
          <li key={post.slug}>
            <a href={`/blog/${post.slug}`}>{post.title}</a>
          </li>
        ))}
      </ul>
    </div>
  );
}

// app/blog/[slug]/page.js

  return (
    <div>

الجزء السابع: تمارين الجلسة السادسة والعشرين

✏️ تمرين 1 — صفحات أساسية15 دقيقة

  • أنشئ مشروع Next.js جديد
  • أضف صفحتي /contact و/services كل منها بملف page.js منفصل
  • أضف روابط لهما في التخطيط العام (layout.js)

✏️ تمرين 2 — مسار ديناميكي20 دقيقة

  • أنشئ قسم /products بصفحة رئيسية تعرض قائمة منتجات (بيانات وهمية)
  • أنشئ /products/[id] تعرض params.id
  • اربط كل منتج في القائمة برابط لصفحته الديناميكية

✏️ تمرين 3 — Server مقابل Client25 دقيقة

  • ابنِ مكوّن عدّاد تفاعلي (Client Component بـ "use client" وuseState)
  • ابنِ مكوّناً آخر يعرض نصاً ثابتاً فقط (Server Component بلا أي تصريح)
  • استخدم الاثنين معاً في نفس الصفحة، ولاحظ أن أحدهما فقط يحتاج "use client"

الواجب: موقع بورتفوليو شخصي بـ Next.js

أنشئ موقع بورتفوليو شخصي كامل باستخدام Next.js وFile-based Routing:

◆ الصفحات المطلوبة:

  • الرئيسية: نبذة ترحيبية قصيرة
  • من أنا: نبذة أطول + المهارات
  • المشاريع: قائمة مشاريع (بيانات وهمية) + صفحة تفاصيل ديناميكية /projects/[id]
  • تواصل: نموذج بسيط (اسم، بريد، رسالة) كـ Client Component

◆ متطلبات تقنية:

  • layout.js واحد مشترك بشريط تنقّل يظهر في كل الصفحات
  • تحديد أي مكوّن يحتاج "use client" بدقّة (فقط ما يستخدم useState أو أحداثاً فعلياً)
  • استخدام metadata في layout.js لعنوان ووصف الموقع
