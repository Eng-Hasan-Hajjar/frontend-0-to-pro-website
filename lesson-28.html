الجزء الأول: التحضير للنشر

1.1

فحص البناء المحلي

Shell 
npm run build
💡 لماذا هذه الخطوة إلزامية

npm run build يُنفِّذ نفس عملية البناء التي ستحدث على خادم النشر — إن نجحت محلياً، فالنشر شبه مضمون النجاح. إن فشلت، من الأفضل معرفة ذلك الآن لا بعد رفع الموقع فعلياً.

1.2

أخطاء شائعة عند البناء

الخطأالسبب الشائعالحل
Module not foundاستيراد ملف بمسار خاطئ أو بحالة أحرف مختلفةتحقّق من المسار وحالة الأحرف بدقة (Linux حسّاس لحالة الأحرف)
'useState' is not definedنسيان "use client" في مكوّن يستخدم Hooksأضف 'use client' في أول سطر من الملف
Hydration Mismatchاختلاف بين ما يُرسَم على الخادم وما يُرسَم في المتصفح (مثل Date.now() مباشرة في JSX)انقل القيم غير الحتمية لتُحسَب داخل useEffect بدل الرسم المباشر

الجزء الثاني: رفع المشروع على GitHub

2.1

التأكد من .gitignore

Shell 
# .gitignore الأساسي (Next.js ينشئه تلقائياً عند إنشاء المشروع)
node_modules/
.next/
.env.local
⚠️ تحذير أمني

لا تسمح أبداً بأن يُرفَع ملف .env.local إلى GitHub — فهو يحوي أسراراً (مفاتيح API، كلمات مرور قواعد بيانات) يجب أن تبقى خاصة تماماً.

2.2

أوامر git الأساسية

Shell 
git init
git add .
git commit -m "أول نسخة من المشروع"
git branch -M main
git remote add origin https://github.com/username/my-app.git
git push -u origin main

الجزء الثالث: النشر على Vercel

Vercel هي الشركة التي تطوّر Next.js نفسه، ولذلك النشر عليها هو الأسهل والأكثر توافقاً بلا أي إعداد إضافي تقريباً.

3.1

خطوات النشر

  • افتح vercel.com وسجّل الدخول بحساب GitHub نفسه
  • اضغط "Add New Project"
  • اختر المستودع (Repository) الذي رفعته في الجزء السابق
  • اترك كل الإعدادات الافتراضية — Vercel يكتشف أنه مشروع Next.js تلقائياً
  • اضغط "Deploy" وانتظر دقيقة إلى دقيقتين
🎉 النتيجة

ستحصل على رابط حقيقي فوراً، بصيغة شبيهة بـ my-app.vercel.app — قابل للمشاركة مع أي شخص في العالم فوراً.

🔐

الجزء الرابع: متغيرات البيئة في الإنتاج

4.1

محلياً مقابل الإنتاج

Shell 
# .env.local (محلياً فقط، لا يُرفَع لـ GitHub أبداً بفضل .gitignore)
NEXT_PUBLIC_API_URL=https://api.example.com
DATABASE_SECRET=my-secret-key
JavaScript 
// الاستخدام داخل الكود
const apiUrl = process.env.NEXT_PUBLIC_API_URL;
💡 قاعدة مهمة

المتغيرات التي تبدأ بـ NEXT_PUBLIC_ تصل إلى متصفح المستخدم (Client)، أما البقية فتبقى على الخادم فقط لأسباب أمنية — لا تضع أسراراً حساسة خلف NEXT_PUBLIC_ أبداً.

4.2

إضافتها في Vercel

  • من لوحة تحكّم المشروع في Vercel: Settings → Environment Variables
  • أضف نفس المفاتيح والقيم الموجودة في .env.local محلياً
  • أعد النشر (Redeploy) بعد أي تعديل على المتغيرات ليأخذ التطبيق القيم الجديدة

الجزء الخامس: مراجعة شاملة — أسئلة وأجوبة

مراجعة الجلسات 17-19 — React الأساسي

السؤالالإجابة الصحيحة
ما الفرق بين State وProps؟State مملوكة للمكوّن نفسه وقابلة للتغيير من الداخل؛ Props تُمرَّر من الأب وللقراءة فقط
متى نحتاج useEffect؟لأي Side Effect: جلب بيانات، مؤقّتات، الاستماع لأحداث المتصفح
ما الفرق بين useEffect(fn) وuseEffect(fn, [])؟الأول بعد كل رسم بلا استثناء؛ الثاني مرة واحدة فقط عند الظهور الأول

مراجعة الجلسات 20-22 — Hooks متقدمة وRouting

السؤالالإجابة الصحيحة
متى نستخدم useRef بدل useState؟عند حاجتنا لتذكّر قيمة أو الوصول لعنصر DOM دون التسبب بإعادة رسم
ما فائدة Context API الأساسية؟مشاركة بيانات عبر أي عمق من المكوّنات دون Prop Drilling
ما الفرق بين Link وNavLink؟كلاهما تنقّل فوري بلا إعادة تحميل، لكن NavLink يُنسِّق الرابط النشط تلقائياً

مراجعة الجلسات 23-25 — Layouts والأداء وuseReducer

السؤالالإجابة الصحيحة
ما فائدة Outlet؟تحديد مكان عرض المسار الفرعي داخل تخطيط (Layout) مشترك
متى يستحق React.memo الاستخدام فعلياً؟فقط لمكوّنات ثقيلة، بعد رصد مشكلة أداء حقيقية مُقاسة — لا كتحسين افتراضي
متى نفضّل useReducer على useState؟عند حالة معقّدة بعدة عمليات تحديث مترابطة يصعب تتبّعها بدوال useState منفصلة

مراجعة الجلستين 26-27 — Next.js

السؤالالإجابة الصحيحة
ما الفرق بين Server وClient Component؟الأول (الافتراضي) يعمل على الخادم فقط؛ الثاني يحتاج "use client" لأي تفاعل (State، أحداث)
كيف نُنشئ API Route؟ملف route.js داخل app/api/يُصدِّر دوال GET/POST وغيرها
ما فائدة File-based Routing؟بنية مجلدات app/ تُحدِّد مسارات الموقع تلقائياً، بلا إعداد Router يدوي

الجزء السادس: المشروع الختامي — انشر تطبيقك فعلياً

خُذ أي مشروع Next.js بنيته في الجلستين 26 أو 27 (أو مشروع البورتفوليو من واجب الجلسة 26)، ونفِّذ خطوات النشر كاملة عليه:

  • 1) نفِّذ npm run build محلياً وتأكد من عدم وجود أخطاء
  • 2) ارفع المشروع على GitHub بأوامر git كاملة
  • 3) اربطه بـ Vercel وانشره
  • 4) إن كان يستخدم متغيرات بيئة، أضفها في إعدادات Vercel وأعد النشر
  • 5) افتح الرابط الحقيقي وتأكد أن كل الصفحات والـ API Routes تعمل كما هو متوقَّع

المشروع: منصة مدونة تقنية متكاملة — منشورة فعلياً

هذا مشروع التخرّج الكامل لمرحلة React وNext.js — يجب أن يجمع كل ما تعلمناه من الجلسة 17 حتى 28:

◆ المتطلبات الوظيفية:

  • صفحة رئيسية وصفحة "من نحن" وقسم مدونة كامل (File-based Routing)
  • صفحة مقال فردي بمسار ديناميكي [slug]
  • API Routes لجلب المقالات (GET) وإضافة مقال جديد (POST)
  • نموذج تواصل كـ Client Component يُرسل بيانات لـ API Route داخلي
  • إدارة حالة مشتركة (Context أو useReducer) لعنصر واحد على الأقل (مثل وضع Theme، أو المفضّلة)

◆ متطلبات تقنية إلزامية:

  • التمييز الصحيح بين Server وClient Components في كل الملفات
  • npm run build ناجح تماماً بلا أخطاء
  • مشروع مرفوع على GitHub بـ .gitignore صحيح (لا أسرار مرفوعة)
  • منشور فعلياً على Vercel برابط حقيقي يعمل
