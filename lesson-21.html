الجلسة الحادية والعشرون — useContext وإدارة الحالة المشتركة

Prop Drilling • Context API • createContext • Provider • useContext • نظام Theme كامل

👤 المؤلف: Eng. Hasan Hajjar  ·  Software Engineer & AI Engineer

🎯 أهداف الجلسة الحادية والعشرين

  • فهم مشكلة Prop Drilling ولماذا تُصبح مزعجة في التطبيقات المتوسطة والكبيرة
  • استخدام Context API: createContext، Provider، useContext
  • بناء Context كامل لمشاركة بيانات عبر أي عمق من المكوّنات دون تمرير Props
  • دمج useContext مع useState لإدارة حالة مشتركة قابلة للتحديث من أي مكوّن
  • تنظيم Context في ملف منفصل مع Custom Hook مخصَّص لتسهيل استخدامه
  • معرفة متى Context هو الحل الصحيح، ومتى تكفي Props وLifting State Up
  • بناء نظام Theme (فاتح/داكن) كامل يعمل عبر التطبيق بالكامل بلا أي Prop Drilling
🔁 مراجعة سريعة من الجلسة العشرين

useRef يحفظ قيمة بين الرسومات دون التسبب بإعادة رسم، ويُستخدَم للوصول المباشر لعناصر DOM

useState لما يجب أن يظهر في الواجهة، useRef لما لا يحتاج عرضاً

Custom Hook دالة تبدأ بـ use، تستخدم Hooks أخرى بداخلها، لاستخراج منطق متكرر

useFetch وuseLocalStorage أمثلة على Custom Hooks عملية قابلة لإعادة الاستخدام

🎓 أهلاً بحل مشكلة تمرير البيانات!

لاحظت أن Props تُمرَّر أحياناً عبر مكوّنات وسيطة لا تستخدمها إطلاقاً، فقط لتصل لمكوّن عميق بحاجتها فعلاً؟

هذا ما يُسمّى Prop Drilling، واليوم نتعلّم Context API — الحل الرسمي في React لمشاركة بيانات عبر أي عمق من الشجرة دون تمريرها يدوياً عبر كل مستوى.

🕳️

الجزء الأول: مشكلة Prop Drilling

1.1

مثال: تمرير عبر 4 طبقات

JSX 
function App() {
  const [user, setUser] = useState({ name: 'سارة', role: 'admin' });
  return <Layout user={user} />;
}

function Layout({ user }) {
  return (
    <div>
      <Header user={user} />
      <MainContent user={user} />
    </div>
  );
}

function Header({ user }) {
  return <Toolbar user={user} />; // Header لا يستخدم user إطلاقاً — يُمرِّره فقط!
}

function Toolbar({ user }) {
  return <UserAvatar user={user} />; // Toolbar أيضاً لا يستخدمه!
}

function UserAvatar({ user }) {
  return <span>{user.name}</span>; // أخيراً هنا فعلاً نحتاج user
}
1.2

لماذا هذا مزعج؟

  • مكوّنان وسيطان (Header، Toolbar) "يعرفان" عن user رغم عدم استخدامه فيهما إطلاقاً
  • أي تعديل على شكل بيانات user يتطلّب تعديل توقيع كل مكوّن في السلسلة
  • كلما كبر التطبيق، زادت طبقات التمرير، وأصبح تتبّع مصدر البيانات أصعب

الجزء الثاني: Context API — الأساسيات

2.1

الخطوات الثلاث

JSX 
import { createContext, useContext } from 'react';

// 1) إنشاء الـ Context
const UserContext = createContext(null);

// 2) تزويد القيمة عبر Provider في الأعلى
function App() {
  const user = { name: 'سارة', role: 'admin' };
  return (
    <UserContext.Provider value={user}>
      <Layout />
    </UserContext.Provider>
  );
}

// 3) أي مكوّن بأي عمق يقرأ القيمة مباشرة، بلا Props إطلاقاً!
function Layout() {
  return (
    <div>
      <Header />
      <MainContent />
    </div>
  );
}

function Header() {
  return <Toolbar />; // لا حاجة لتمرير أي شيء!
}

function Toolbar() {
  return <UserAvatar />;
}

function UserAvatar() {
  const user = useContext(UserContext); // القراءة المباشرة من أي عمق
  return <span>{user.name}</span>;
}
💡 الفكرة الجوهرية

Provider يُحيط جزءاً من الشجرة ويُزوِّده بقيمة معيّنة. أي مكوّن داخل هذا الجزء — مهما كان عمقه — يستطيع قراءة القيمة مباشرة عبر useContext، بلا أي تمرير يدوي.

🎨

الجزء الثالث: بناء نظام Theme كامل

3.1

إنشاء ThemeContext مع State

JSX 
// context/ThemeContext.js
import { createContext, useState } from 'react';

const ThemeContext = createContext(null);

function ThemeProvider({ children }) {
  const [theme, setTheme] = useState('light');

  function toggleTheme() {
    setTheme((prev) => (prev === 'light' ? 'dark' : 'light'));
  }

  return (
    <ThemeContext.Provider value={{ theme, toggleTheme }}>
      {children}
    </ThemeContext.Provider>
  );
}

export { ThemeContext, ThemeProvider };
3.2

استخدامه في مكوّنات عميقة بلا أي Props

JSX 
// src/main.jsx
import { ThemeProvider } from './context/ThemeContext';

ReactDOM.createRoot(document.getElementById('root')).render(
  <ThemeProvider>
    <App />
  </ThemeProvider>
);

الجزء الرابع: دمج useContext مع useState

من الشائع بناء Custom Hook فوق useContext مباشرة، لتسهيل الاستخدام وإضافة رسالة خطأ واضحة إن استُخدم خارج الـ Provider.

4.1

Custom Hook: useTheme

React 
// context/ThemeContext.js (إضافة)
import { useContext } from 'react';

function useTheme() {
  const context = useContext(ThemeContext);
  if (!context) {
    throw new Error('useTheme يجب أن يُستخدَم داخل ThemeProvider');
  }
  return context;
}

export { ThemeContext, ThemeProvider, useTheme };
4.2

استخدامه في أي مكوّن

JSX 
function ThemeToggleButton() {
  const { theme, toggleTheme } = useTheme();

  return (
    <button onClick={toggleTheme}>
      {theme === 'light' ? '🌙 التبديل للوضع الداكن' : '☀️ التبديل للوضع الفاتح'}
    </button>
  );
}

function Card() {
  const { theme } = useTheme();
  return (
    <div className={theme === 'dark' ? 'card-dark' : 'card-light'}>
      محتوى البطاقة
    </div>
  );
}
⭐ لماذا هذا أفضل

أي مكوّن في التطبيق يستطيع الوصول لـ theme وtoggleTheme بسطر واحد (useTheme())، بلا استيراد useContext يدوياً في كل مرة، وبرسالة خطأ واضحة إن نسي أحدهم لفّ التطبيق بـ ThemeProvider.

⚖️

الجزء الخامس: متى نستخدم Context ومتى تكفي Props؟

السؤالالقرار
هل البيانات تُستخدَم في مكوّنين أو ثلاثة قريبين فقط؟✅ مرّرها بـ Props مباشرة
هل يحتاجها مكوّنون منتشرون بأعماق مختلفة (Theme، المستخدم، اللغة)؟🌳 استخدم Context
هل تُمرَّر عبر أكثر من 2-3 مكوّنات وسيطة لا تستخدمها؟🌳 علامة واضحة لاستخدام Context
هل تتغيّر البيانات بسرعة كبيرة جداً (حركة الفأرة مثلاً)؟⚠️ تجنّب Context — قد يُسبِّب إعادة رسم واسعة غير ضرورية
💻

الجزء السادس: المشروع — تطبيق بنظام Theme متكامل

تطبيق بسيط بمكوّنين مختلفين (Navbar وDashboard) يتشاركان حالة الوضع الليلي/النهاري بالكامل عبر Context، بلا أي Prop Drilling.

src/context/ThemeContext.jsx

JSX 
import { createContext, useContext, useState } from 'react';

const ThemeContext = createContext(null);

export function ThemeProvider({ children }) {
  const [theme, setTheme] = useState('light');
  const toggleTheme = () => setTheme((t) => (t === 'light' ? 'dark' : 'light'));

  return (
    <ThemeContext.Provider value={{ theme, toggleTheme }}>
      <div className={`app-${theme}`}>{children}</div>
    </ThemeContext.Provider>
  );
}

export function useTheme() {
  const ctx = useContext(ThemeContext);
  if (!ctx) throw new Error('useTheme يجب أن يُستخدَم داخل ThemeProvider');
  return ctx;
}

src/App.jsx

JSX 
import { ThemeProvider } from './context/ThemeContext';
import Navbar from './components/Navbar';
import Dashboard from './components/Dashboard';

function App() {
  return (
    <ThemeProvider>
      <Navbar />
      <Dashboard />
    </ThemeProvider>
  );
}

export default App;

src/components/Navbar.jsx و Dashboard.jsx

JSX 
// src/components/Navbar.jsx
import { useTheme } from '../context/ThemeContext';

function Navbar() {
  const { theme, toggleTheme } = useTheme();
  return (
    <nav>
      <h1>لوحتي</h1>
      <button onClick={toggleTheme}>{theme === 'light' ? '🌙' : '☀️'}</button>
    </nav>
  );
}

export default Navbar;

// src/components/Dashboard.jsx
import { useTheme } from '../context/ThemeContext';

function Dashboard() {
  const { theme } = useTheme();
  return (
    <main className={`dashboard dashboard-${theme}`}>
      <p>الوضع الحالي: {theme === 'light' ? 'فاتح ☀️' : 'داكن 🌙'}</p>
    </main>
  );
}

export default Dashboard;
💡 لاحظ

لا Navbar ولا Dashboard استقبلا theme كـ Prop من App — كلاهما استدعى useTheme() مباشرة، رغم كونهما في فروع مختلفة تماماً من الشجرة.

الواجب: متجر إلكتروني مصغّر بـ Theme وCart معاً

أنشئ تطبيق React بـ Vite لمتجر إلكتروني بسيط، يجمع Context Providerين معاً بلا أي Prop Drilling:

◆ المتطلبات الوظيفية:

  • ThemeContext: وضع فاتح/داكن يعمل عبر كل صفحات التطبيق
  • CartContext: عربة تسوّق (إضافة، حذف، تعديل الكمية، حساب الإجمالي)
  • صفحة منتجات (ProductsGrid) تعرض المنتجات مع زر "أضف للعربة"
  • Navbar يعرض عدد عناصر العربة وزر تبديل الوضع معاً

◆ متطلبات تقنية:

  • كل Context في ملف منفصل داخل src/context/، مع Custom Hook خاص به
  • لا تمرير Props يدوي لبيانات الـ Theme أو الـ Cart بين أي مكوّنين
  • استخدام useLocalStorage (من الجلسة السابقة) لحفظ الوضع (فاتح/داكن) بعد إعادة تحميل الصفحة
